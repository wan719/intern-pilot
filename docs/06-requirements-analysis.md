# InternPilot 需求分析文档

## 1. 文档说明

本文档用于描述 InternPilot 项目的功能需求、非功能需求、权限需求、数据需求、接口需求和验收标准。

InternPilot 是一个面向大学生实习求职场景的 AI 实习投递与简历优化平台，主要帮助用户完成简历上传解析、岗位 JD 管理、AI 匹配分析、简历优化建议生成和投递记录管理。

---

## 2. 项目需求概述

### 2.1 项目目标

InternPilot 的核心目标是：

> 帮助大学生通过 AI 分析简历与岗位 JD 的匹配程度，发现简历短板，生成优化建议，并系统管理实习投递流程。

系统第一阶段需要完成以下核心闭环：

```text
用户注册登录
  ↓
上传简历
  ↓
录入岗位 JD
  ↓
AI 分析简历与岗位匹配度
  ↓
生成简历优化建议
  ↓
创建投递记录
  ↓
跟踪投递状态
```

### 2.2 目标用户

| 用户类型 | 说明 | 核心需求 |
|---|---|---|
| 普通用户 | 正在准备实习投递的大学生 | 上传简历、分析岗位、管理投递 |
| 管理员 | 系统维护人员 | 管理用户、查看系统数据、维护配置 |

第一阶段主要实现普通用户功能，管理员功能可以作为后续扩展。

### 2.3 系统角色

系统第一阶段包含两个角色：

| 角色 | 说明 |
|---|---|
| USER | 普通用户 |
| ADMIN | 系统管理员 |

## 4. 用户认证模块需求

### 4.1 用户注册

#### 4.1.1 功能描述

用户可以通过用户名、密码、邮箱等信息注册账号。

#### 4.1.2 输入字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| username | String | 是 | 用户名，唯一 |
| password | String | 是 | 密码 |
| confirmPassword | String | 是 | 确认密码 |
| email | String | 否 | 邮箱 |
| school | String | 否 | 学校 |
| major | String | 否 | 专业 |
| grade | String | 否 | 年级 |

#### 4.1.3 业务规则

1. 用户名不能为空；
2. 用户名不能重复；
3. 密码不能为空；
4. 两次密码必须一致；
5. 密码需要使用 BCrypt 加密后存储；
6. 邮箱如果填写，需要符合邮箱格式；
7. 注册成功后默认角色为 USER；
8. 注册成功后返回用户基础信息，不返回密码。

#### 4.1.4 异常情况

| 异常 | 返回说明 |
|---|---|
| 用户名为空 | 用户名不能为空 |
| 用户名重复 | 用户名已存在 |
| 密码为空 | 密码不能为空 |
| 两次密码不一致 | 两次密码输入不一致 |
| 邮箱格式错误 | 邮箱格式不正确 |

#### 4.1.5 验收标准

1. 用户可以正常注册；
2. 数据库中密码不是明文；
3. 重复用户名不能注册；
4. 注册成功后数据库存在用户记录；
5. 返回结果不包含密码字段。

### 4.2 用户登录

#### 4.2.1 功能描述

用户可以使用用户名和密码登录系统，登录成功后系统返回 JWT Token。

#### 4.2.2 输入字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| username | String | 是 | 用户名 |
| password | String | 是 | 密码 |

#### 4.2.3 输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| token | String | JWT Token |
| tokenType | String | Bearer |
| userId | Long | 用户 ID |
| username | String | 用户名 |
| role | String | 用户角色 |

#### 4.2.4 业务规则

1. 用户名和密码不能为空；
2. 用户不存在时登录失败；
3. 密码错误时登录失败；
4. 登录成功后生成 JWT；
5. JWT 中应包含 userId、username、role 等信息；
6. 前端后续访问接口时需要携带 Authorization Header。

#### 4.2.5 验收标准

1. 正确用户名和密码可以登录；
2. 错误密码不能登录；
3. 登录成功返回 Token；
4. 携带 Token 可以访问受保护接口；
5. 不携带 Token 访问业务接口返回 401。

### 4.3 退出登录

#### 4.3.1 功能描述

第一阶段可以由前端删除本地 Token 实现退出登录。

#### 4.3.2 后续扩展

后续可以引入 Redis Token 黑名单，实现服务端主动失效 Token。

|---|---|---|

| school | String | 学校 |
| major | String | 专业 |

| role | String | 角色 |

#### 5.1.5 验收标准

用户可以修改自己的邮箱、学校、专业、年级等信息。

#### 5.2.2 可修改字段

| 字段 | 类型 | 是否必填 |

| school | String | 否 |
| major | String | 否 |
| grade | String | 否 |

#### 5.2.3 业务规则

1. 用户只能修改自己的信息；
2. 用户名第一阶段不允许修改；
3. 角色不允许普通用户自行修改；

#### 5.2.4 验收标准

2. 修改后再次查询能看到新数据；
3. 普通用户不能修改 role 字段。

## 6. 简历管理模块需求

### 6.1 上传简历

#### 6.1.1 功能描述

用户可以上传 PDF 或 DOCX 格式的简历，系统保存文件并解析简历文本。

#### 6.1.2 权限要求

必须登录。

#### 6.1.3 支持格式

| 文件类型 | 是否支持 | 解析方式 |
|---|---|---|
| PDF | 是 | Apache PDFBox |
| DOCX | 是 | Apache POI |
| TXT | 可选 | 直接读取文本 |
| Markdown | 可选 | 直接读取文本 |

第一阶段必须支持 PDF 和 DOCX。

#### 6.1.4 输入字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| file | MultipartFile | 是 | 简历文件 |
| resumeName | String | 否 | 简历名称 |

#### 6.1.5 输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| resumeId | Long | 简历 ID |
| fileName | String | 文件名 |
| fileType | String | 文件类型 |
| parsedText | String | 解析后的简历文本 |
| createdAt | DateTime | 上传时间 |

#### 6.1.6 业务规则

1. 用户必须登录后才能上传简历；
2. 文件不能为空；
3. 文件类型必须是 PDF 或 DOCX；
4. 文件大小需要限制，建议最大 10MB；
5. 文件保存到本地目录或对象存储；
6. 系统解析文件文本并保存到数据库；
7. 每份简历必须绑定当前用户 ID；
8. 用户只能查看和操作自己的简历。

#### 6.1.7 异常情况

| 异常 | 返回说明 |
|---|---|
| 文件为空 | 上传文件不能为空 |
| 文件过大 | 文件大小不能超过限制 |
| 文件格式不支持 | 仅支持 PDF/DOCX 文件 |
| 解析失败 | 简历解析失败，请检查文件格式 |
| 未登录 | 请先登录 |

#### 6.1.8 验收标准

1. 用户可以上传 PDF 简历；
2. 用户可以上传 DOCX 简历；
3. 数据库保存简历记录；
4. 数据库保存解析后的文本；
5. 非支持格式上传失败；
6. 未登录上传失败。

### 6.2 查询简历列表

#### 6.2.1 功能描述

用户可以查看自己上传过的简历列表。

#### 6.2.2 权限要求

必须登录。

#### 6.2.3 输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | Long | 简历 ID |
| fileName | String | 文件名 |
| fileType | String | 文件类型 |
| isDefault | Boolean | 是否默认简历 |
| createdAt | DateTime | 上传时间 |

#### 6.2.4 业务规则

1. 只返回当前用户的简历；
2. 默认按上传时间倒序；
3. 列表接口不返回完整 parsedText，避免响应过大；
4. parsedText 通过详情接口查询。

#### 6.2.5 验收标准

1. 用户只能看到自己的简历；
2. 简历列表按时间倒序；
3. 列表数据不包含其他用户简历。

### 6.3 查询简历详情

#### 6.3.1 功能描述

用户可以查看某份简历的详细信息和解析文本。

#### 6.3.2 输入参数

| 参数 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| resumeId | Long | 是 | 简历 ID |

#### 6.3.3 业务规则

1. 简历必须存在；
2. 当前用户必须是简历所有者；
3. 返回解析文本 parsedText；
4. 如果不是自己的简历，返回 403 或 404。

#### 6.3.4 验收标准

1. 用户可以查看自己的简历详情；
2. 用户不能查看他人的简历详情；
3. 简历不存在时返回错误。

### 6.4 删除简历

#### 6.4.1 功能描述

用户可以删除不需要的简历。

#### 6.4.2 业务规则

1. 只能删除自己的简历；
2. 第一阶段可以逻辑删除；
3. 如果简历有关联分析报告，建议保留历史报告；
4. 删除默认简历后，需要清空默认状态；
5. 后续可扩展删除本地文件。

#### 6.4.3 验收标准

1. 用户可以删除自己的简历；
2. 删除后列表不再显示；
3. 用户不能删除他人的简历。

### 6.5 设置默认简历

#### 6.5.1 功能描述

用户可以将一份简历设置为默认简历。

#### 6.5.2 业务规则

1. 每个用户只能有一份默认简历；
2. 设置新默认简历时，取消旧默认简历；
3. 只能设置自己的简历为默认简历。

#### 6.5.3 验收标准

1. 用户可以设置默认简历；
2. 同一用户始终只有一份默认简历；
3. 不能设置他人简历为默认简历。

### 7.1 创建岗位 JD

#### 7.1.1 功能描述

用户可以创建目标实习岗位 JD，用于后续 AI 匹配分析。

#### 7.1.2 权限要求

必须登录。

#### 7.1.3 输入字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| companyName | String | 是 | 公司名称 |
| jobTitle | String | 是 | 岗位名称 |
| jobType | String | 否 | 岗位类型 |
| location | String | 否 | 工作地点 |
| sourcePlatform | String | 否 | 来源平台 |
| jobUrl | String | 否 | 岗位链接 |
| jdContent | String | 是 | 岗位 JD 内容 |

#### 7.1.4 业务规则

2. 岗位名称不能为空；
3. 岗位 JD 内容不能为空；
4. 岗位必须绑定当前用户；

| 条件 | 类型 | 说明 |
|---|---|---|

| jobType | String | 岗位类型 |
| location | String | 工作地点 |
| pageNum | Integer | 页码 |
| pageSize | Integer | 每页数量 |

#### 7.2.3 业务规则

1. 只查询当前用户创建的岗位；
2. 支持分页；
3. 支持关键词搜索；

用户可以查看某个岗位 JD 的完整详情。

#### 7.3.2 业务规则

1. 岗位必须存在；
2. 当前用户必须是岗位所有者；
3. 返回完整 jdContent；
4. 可展示该岗位是否已有分析报告或投递记录。

#### 7.3.3 验收标准

1. 用户可以查看自己的岗位详情；
2. 用户不能查看他人的岗位详情。

### 7.4 修改岗位 JD

#### 7.4.1 功能描述

#### 7.4.2 业务规则

### 7.5 删除岗位 JD

2. 如果岗位已有投递记录，建议逻辑删除；
3. 删除岗位后不再出现在岗位列表中；
4. 历史分析报告可以保留。

#### 7.5.3 验收标准

1. 用户可以删除自己的岗位；
2. 删除后列表不显示；
3. 用户不能删除他人的岗位。

## 8. AI 分析模块需求

### 8.1 简历与岗位匹配分析

#### 8.1.1 功能描述

用户选择一份简历和一个岗位 JD，系统调用大语言模型，分析简历与岗位之间的匹配程度。

这是系统的核心功能。

#### 8.1.2 权限要求

必须登录。

#### 8.1.3 输入字段

| 字段 | 类型 | 是否必填 | 说明 |
|---|---|---|---|
| resumeId | Long | 是 | 简历 ID |
| jobId | Long | 是 | 岗位 ID |

#### 8.1.4 输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| reportId | Long | 分析报告 ID |
| matchScore | Integer | 匹配分数，0-100 |
| matchLevel | String | 匹配等级 |
| strengths | List/String | 简历优势 |
| weaknesses | List/String | 简历短板 |
| missingSkills | List/String | 缺失技能 |
| suggestions | List/String | 简历优化建议 |
| interviewTips | List/String | 面试准备建议 |
| createdAt | DateTime | 分析时间 |

#### 8.1.5 业务流程

```text
用户选择简历和岗位
  ↓
后端校验简历是否属于当前用户
  ↓
后端校验岗位是否属于当前用户
  ↓
检查 Redis 缓存
  ↓
缓存命中：直接返回分析结果
  ↓
缓存未命中：构造 Prompt
  ↓
调用 AI API
  ↓
解析 AI 返回内容
  ↓
保存分析报告到数据库
  ↓
写入 Redis 缓存
  ↓
返回分析报告
```

#### 8.1.6 业务规则

1. resumeId 不能为空；
2. jobId 不能为空；
3. 简历必须属于当前用户；
4. 岗位必须属于当前用户；
5. 简历 parsedText 不能为空；
6. 岗位 jdContent 不能为空；
7. AI 返回结果需要进行格式校验；
8. 分析报告必须保存到数据库；
9. AI 原始返回内容 rawResponse 需要保存；
10. 同一用户、同一简历、同一岗位的分析结果可以缓存。

#### 8.1.7 匹配等级规则

| 分数范围 | 匹配等级 |
|---|---|
| 85-100 | HIGH |
| 70-84 | MEDIUM_HIGH |
| 60-69 | MEDIUM |
| 40-59 | LOW |
| 0-39 | VERY_LOW |

#### 8.1.8 异常情况

| 异常 | 返回说明 |
|---|---|
| 简历不存在 | 简历不存在 |
| 岗位不存在 | 岗位不存在 |
| 无权访问简历 | 无权访问该简历 |
| 无权访问岗位 | 无权访问该岗位 |
| 简历文本为空 | 简历解析文本为空 |

## 10. Redis 缓存需求

### 10.1 AI 分析结果缓存

#### 10.1.1 功能描述

为减少重复调用 AI API，系统需要缓存相同简历和岗位组合的分析结果。

#### 10.1.2 缓存 Key 设计

建议格式：

```text
internpilot:analysis:{userId}:{resumeId}:{jobId}
```

#### 10.1.3 缓存 Value

可以存储分析报告 JSON。

#### 10.1.4 缓存过期时间

建议第一阶段设置为 24 小时，也可以后续调整为 7 天。

#### 10.1.5 业务规则

1. 调用 AI 前先查 Redis；
2. 命中缓存则直接返回；
3. 未命中才调用 AI；
4. AI 分析成功后写入 Redis；
5. 简历或岗位内容修改后，应考虑清理旧缓存。

#### 10.1.6 验收标准

1. 第一次分析调用 AI；
2. 第二次相同简历和岗位分析命中缓存；
3. 缓存命中时响应速度明显更快；
4. Redis 不可用时不能影响核心业务，可以降级为直接调用 AI。

7. 使用统一响应结构；
8. 使用统一异常处理；
9. 使用枚举管理状态值；

## 18. 需求分析结论

InternPilot 第一阶段需求聚焦于“简历上传解析、岗位 JD 管理、AI 匹配分析、投递记录管理”四个核心业务模块，并通过用户认证和权限控制保证数据安全。

从业务角度看，系统可以解决大学生实习投递中简历优化缺乏针对性、岗位匹配判断困难、投递记录分散等问题。

从技术角度看，系统需求适合使用 Spring Boot、Spring Security、JWT、MySQL、Redis、文件解析和大语言模型 API 等技术实现，既能体现 Java 后端能力，也能体现 AI 应用集成能力。

第一阶段建议严格控制范围，优先完成 MVP 核心闭环，后续再逐步增加 WebSocket、RAG、岗位推荐、数据看板和管理员后台等高级功能。
