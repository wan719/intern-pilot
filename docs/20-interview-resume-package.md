## 3. 简历 Bullet 推荐版

- 基于 Spring Security + JWT 实现用户注册登录、无状态认证和接口鉴权，统一处理未登录、Token 失效和权限不足场景。

- 设计岗位 JD、AI 分析报告和投递记录模块，支持岗位搜索、匹配分析、报告查询、投递状态跟踪和复盘备注。
- 集成大语言模型 API，通过 Prompt Engineering 生成匹配分数、优劣势、技能缺口、优化建议和面试准备建议。
- 使用 Redis 缓存相同用户、简历、岗位组合的 AI 分析结果，减少重复模型调用并提升接口响应速度。
- 基于 userId 实现简历、岗位、分析报告和投递记录的数据隔离，防止水平越权访问。
- 通过统一响应结构、全局异常处理、参数校验、Swagger 文档和 Docker Compose 提升项目工程化程度。

## 4. 1 分钟项目介绍

我的项目叫 InternPilot，是一个面向大学生实习求职的 AI 简历岗位匹配和投递管理平台。

用户可以上传 PDF 或 DOCX 简历，系统会解析出简历文本；然后用户可以保存目标岗位 JD。之后系统会把简历文本和岗位 JD 组合成 Prompt，调用大语言模型分析匹配度，生成匹配分数、优势、短板、技能缺口、简历优化建议和面试准备建议。

除了 AI 分析，我还做了投递记录管理，用户可以记录岗位处于待投递、已投递、笔试、面试、Offer 或被拒等状态，形成完整的实习投递闭环。

技术上主要使用 Spring Boot 3、Spring Security、JWT、MyBatis Plus、MySQL、Redis、PDFBox、Apache POI 和大语言模型 API。项目重点实现了 JWT 鉴权、数据权限控制、文件解析、AI 结果缓存和统一异常处理。

## 5. 3 分钟项目介绍

InternPilot 是我为了提升大学生找实习效率做的项目。它主要解决三个问题：简历和岗位是否匹配、简历应该如何优化、投递过程如何管理。

系统后端基于 Spring Boot 3 构建。用户注册登录后，后端签发 JWT，后续请求通过 Authorization Header 携带 Token。Spring Security 中配置 JWT 过滤器，负责解析 Token，并把用户信息放入 SecurityContext。

业务流程上，用户先上传 PDF 或 DOCX 简历。后端会校验文件大小、扩展名和 MIME 类型，然后保存文件，并使用 PDFBox 或 Apache POI 提取文本，保存到 `resume.parsed_text`。用户再创建岗位 JD，保存公司、岗位、地点、薪资和完整 JD 内容。

AI 分析时，用户选择一份简历和一个岗位。后端先校验这两个资源都属于当前用户，避免水平越权；然后读取简历文本和岗位 JD，构造 Prompt 调用大语言模型。Prompt 要求模型严格返回 JSON，字段包括 `matchScore`、`matchLevel`、`strengths`、`weaknesses`、`missingSkills`、`suggestions` 和 `interviewTips`。后端解析 JSON 后保存到 `analysis_report` 表，并返回给前端。

为了减少重复 AI 调用，我使用 Redis 缓存相同 `userId + resumeId + jobId` 组合的分析结果。如果用户再次分析同一组合，就直接返回缓存；如果传入 `forceRefresh=true`，才重新调用 AI。

最后，系统支持投递记录管理。用户可以基于岗位、简历和分析报告创建投递记录，并更新状态、面试时间、备注和复盘内容。这个模块让系统从单纯的 AI 分析工具变成完整的求职管理工具。

## 6. 面试重点亮点

### Spring Security + JWT

可以重点讲：

- 登录成功后签发 JWT，后续请求通过 Bearer Token 访问接口。
- JWT 中包含 userId、username 和 role。
- JwtAuthenticationFilter 解析 Token，并写入 SecurityContext。
- 未登录或 Token 无效返回 401，权限不足返回 403。

### 数据权限控制

可以重点讲：

- JWT 只能证明“你是谁”，不能证明“这条数据是不是你的”。
- 简历、岗位、报告、投递记录都绑定 userId。
- 查询详情、修改、删除时同时校验资源 ID 和当前用户 ID。

典型查询思路：

```sql
WHERE id = ? AND user_id = 当前登录用户ID AND deleted = 0
```

### 文件上传与解析

可以重点讲：

- 不直接使用用户原始文件名，避免重名和路径安全问题。
- 存储文件名由 userId、时间戳和随机串生成。
- PDF 使用 PDFBox 解析，DOCX 使用 Apache POI 解析。
- 扫描版 PDF 暂时无法直接解析，后续可接 OCR。

### AI 分析

可以重点讲：

- Prompt 明确要求模型返回 JSON，避免自然语言结果难以入库。
- 后端会清理 Markdown 代码块，只截取 JSON 主体。
- 解析成功后保存结构化报告，方便历史查询和投递记录关联。
- 保留 raw_ai_response，便于排查模型输出异常。

### Redis 缓存

可以重点讲：

- AI 调用比普通查询慢，也可能产生调用成本。
- 相同用户、相同简历、相同岗位短时间内重复分析没有必要。
- 缓存 Key 为 `internpilot:analysis:{userId}:{resumeId}:{jobId}`。
- `forceRefresh=true` 可绕过缓存重新分析。

## 7. 高频追问回答

### 为什么做这个项目？

因为大学生投递实习时经常不知道简历和岗位是否匹配，也不知道简历应该针对某个岗位如何优化。现在大语言模型可以理解简历和 JD，所以我把 AI 分析能力和投递管理结合起来，做成一个从简历分析到投递跟踪的完整工具。

### 和普通 CRUD 项目有什么区别？

它不只是增删改查。项目里有认证鉴权、文件上传解析、AI 调用、Redis 缓存、用户数据隔离和完整业务闭环。尤其是 AI 分析和投递记录关联后，可以把“分析结果”和“实际投递结果”联系起来，后续还能做数据统计。

### AI 返回格式不稳定怎么办？

我从三层处理：第一，在 Prompt 中明确要求只返回 JSON；第二，后端清理可能出现的 Markdown 代码块，并截取 `{}` 中的 JSON；第三，解析失败时抛出统一 AI 异常，并保留原始响应，方便排查。

### 为什么用 Redis 缓存？

AI 分析接口调用外部模型，响应慢且可能有成本。对同一个用户、同一份简历、同一个岗位，短时间内重复分析通常没有必要，所以使用 Redis 缓存结果。用户需要重新分析时可以通过 `forceRefresh` 主动绕过缓存。

### 为什么要逻辑删除？

简历、岗位、分析报告和投递记录之间存在业务关联。直接物理删除可能让历史报告和投递记录丢失上下文。逻辑删除可以保留历史数据，列表查询时只查 `deleted=0`，后续也方便恢复、审计和统计。

### 项目最难的地方是什么？

最难的是 AI 分析模块的稳定性和工程化处理。模型输出不一定严格稳定，可能带 Markdown 或额外说明，所以需要 Prompt 约束、JSON 提取、异常处理和原始响应保留。同时 AI 调用较慢且有成本，所以还要设计 Redis 缓存和强制刷新机制。

## 8. 演示顺序

1. 打开 README，介绍项目背景和技术栈。
2. 打开 Knife4j：`http://localhost:8080/doc.html`。
3. 注册并登录用户，拿到 JWT。
4. 上传 DOCX 或 PDF 简历，展示解析结果。
5. 创建岗位 JD。
6. 发起 AI 匹配分析，展示分数和建议。
7. 第二次分析同一组合，展示 Redis 缓存命中。
8. 创建投递记录。
9. 更新投递状态和复盘备注。
10. 展示数据库表和 Redis Key。

## 9. 后续优化说法

后续我会从几个方向继续完善：

- 补齐 Vue 3 前端页面，让项目更适合演示。
- AI 分析改成异步任务，返回 taskId 后通过轮询或 WebSocket 查询进度。
- 增加投递状态时间线，记录每次状态变化。
- 增加投递数据看板，统计投递数、面试数、Offer 数和平均匹配分。
- 接入 OCR，支持扫描版 PDF 简历解析。
- 增加 AI 调用日志、成本统计和更精细的缓存失效策略。
- 增加 GitHub Actions 自动测试。

## 10. 最后总结话术

这个项目对我最大的价值是，它把 Java 后端常见能力和 AI 应用场景结合起来了。

我不只是做了用户、简历、岗位这些基础 CRUD，还重点实现了 JWT 鉴权、文件解析、AI Prompt 设计、Redis 缓存、数据权限控制和投递闭环管理。

从业务上看，它解决的是大学生实习投递中的真实问题；从技术上看，它覆盖了 Spring Boot 后端开发中比较核心的认证、安全、缓存、文件处理、数据库建模和工程化能力。
