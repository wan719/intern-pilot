# InternPilot 系统分析文档

## 1. 文档说明

本文档用于对 InternPilot 项目进行系统分析，明确系统的建设背景、目标用户、业务问题、系统角色、业务流程、功能需求、非功能需求、系统边界和后续扩展方向。

InternPilot 是一个面向大学生实习求职场景的 AI 实习投递与简历优化平台，旨在帮助大学生更高效地完成简历优化、岗位匹配分析和实习投递管理。

---

## 2. 系统建设背景

随着大学生实习竞争加剧，越来越多学生需要在大二、大三阶段提前准备暑期实习和日常实习。然而，大多数学生在实习求职过程中存在以下问题：

1. 不清楚自己的简历是否符合岗位要求；
2. 不知道如何根据不同岗位 JD 调整简历内容；
3. 简历中的项目经历表达不够专业；
4. 缺少系统化的投递记录管理；
5. 面试前无法快速根据岗位要求进行针对性准备；
6. 缺少低成本、即时反馈的简历优化工具。

传统方式通常依赖人工修改简历、手动记录投递进度、凭经验判断岗位匹配度。这种方式效率低、反馈慢、针对性不足。随着大语言模型在文本理解、语义分析、内容生成方面能力不断增强，可以将 AI 能力应用到大学生实习求职场景中，帮助用户分析简历与岗位之间的匹配程度，发现技能短板，并生成个性化简历优化建议。

因此，本项目计划建设 InternPilot 系统，构建一个集简历管理、岗位 JD 管理、AI 匹配分析、简历优化建议、投递记录管理于一体的求职辅助平台。

---

## 3. 系统建设目标

InternPilot 的总体目标是：

> 构建一个面向大学生实习求职场景的 AI 实习投递与简历优化平台，帮助用户提升岗位匹配度、优化简历表达、管理投递流程，并为开发者展示完整的 Java 后端工程能力与 AI 应用集成能力。

### 3.1 业务目标

1. 帮助大学生快速判断简历与岗位 JD 的匹配程度；
2. 帮助用户发现简历中的优势、短板和技能缺口；
3. 为不同岗位生成个性化简历优化建议；
4. 帮助用户管理实习投递记录；
5. 帮助用户跟踪投递状态、面试安排和复盘内容；
6. 提高大学生实习投递效率。

### 3.2 技术目标

1. 使用 Spring Boot 构建标准后端项目；
2. 使用 Spring Security + JWT 实现用户认证和接口鉴权；
3. 使用 MySQL 存储用户、简历、岗位、分析报告和投递记录；
4. 使用 Redis 缓存 AI 分析结果，减少重复调用；
5. 使用大语言模型 API 完成简历与岗位 JD 的智能分析；
6. 使用 Swagger / Knife4j 提供接口文档；
7. 使用 Docker Compose 完成基础部署；
8. 形成一个适合 GitHub 展示、简历包装和面试讲解的完整项目。

## 4. 目标用户分析

### 4.1 主要用户

本系统的主要用户是正在准备实习投递的大学生。典型用户画像：

| 用户类型 | 用户特点 | 主要需求 |
|---|---|---|
| 大二学生 | 项目经验较少，刚开始准备实习 | 了解简历短板，明确补强方向 |
| 大三学生 | 准备暑期实习或秋招提前批 | 提高岗位匹配度，管理投递流程 |
| 软件工程学生 | 有课程项目和个人项目 | 优化项目描述，匹配技术岗位 |
| AI/后端方向学生 | 希望投递技术实习 | 分析岗位技能要求，生成准备建议 |

### 4.2 管理员用户

管理员主要负责系统基础管理和内容维护。管理员需求包括：

1. 查看用户数量；
2. 查看简历上传数量；
3. 查看 AI 分析调用情况；
4. 管理异常数据；
5. 管理系统配置；
6. 后续可扩展模型配置、Prompt 模板配置等功能。

## 5. 业务痛点分析

### 5.1 简历与岗位匹配困难

大学生往往不知道自己的简历是否匹配目标岗位。例如，用户投递 Java 后端实习，但简历中没有突出 Spring Boot、MySQL、Redis、接口设计、权限认证等关键词，容易在简历筛选阶段被忽略。

系统需要解决的问题：

1. 提取岗位 JD 中的核心技能要求；
2. 提取简历中的技能和项目经历；
3. 对比简历和岗位之间的匹配程度；
4. 输出匹配分数和技能缺口。

### 5.2 简历优化缺乏针对性

很多学生使用同一份简历投递所有岗位，缺少针对不同岗位的定制化表达。

系统需要解决的问题：

1. 根据岗位 JD 生成定制化优化建议；
2. 指出简历中应该强化的项目经历；
3. 提醒用户补充缺失技能；
4. 帮助用户将普通项目描述改写成更适合招聘筛选的表达。

### 5.3 投递过程缺乏管理

学生通常使用 Excel、备忘录或聊天记录管理投递情况，容易出现信息混乱、状态遗漏、面试时间忘记等问题。

系统需要解决的问题：

1. 统一管理投递岗位；
2. 记录投递状态；
3. 记录笔试、面试和 Offer 情况；
4. 支持备注和复盘；
5. 提供投递数据统计。

### 5.4 面试准备不系统

用户收到面试通知后，往往不知道应该根据岗位要求复习哪些知识点。

系统后续可以解决的问题：

1. 根据岗位 JD 生成面试问题；
2. 根据简历项目生成项目追问；
3. 根据技能缺口生成学习建议；
4. 根据投递状态生成准备清单。

## 6. 系统角色分析

InternPilot 第一阶段主要包含两个角色：

1. 普通用户；
2. 系统管理员。

### 6.1 普通用户

普通用户是系统核心使用者，主要操作包括：

1. 注册账号；
2. 登录系统；
3. 上传简历；
4. 查看简历解析结果；
5. 创建岗位 JD；
6. 发起 AI 匹配分析；
7. 查看分析报告；
8. 创建投递记录；
9. 修改投递状态；
10. 查看个人投递数据。

### 6.2 系统管理员

管理员主要负责系统维护，第一阶段可以不重点实现，后续扩展。管理员操作包括：

1. 查看用户列表；
2. 查看系统数据统计；
3. 查看 AI 调用记录；
4. 管理异常分析结果；
5. 管理系统配置；
6. 管理 Prompt 模板；
7. 管理模型配置。

## 7. 系统业务流程分析

### 7.1 用户注册登录流程

```text
用户进入系统
  ↓
注册账号
  ↓
填写用户名、密码、邮箱等信息
  ↓
系统校验参数
  ↓
密码加密后保存到数据库
  ↓
注册成功
  ↓
用户登录
  ↓
系统校验用户名和密码
  ↓
生成 JWT Token
  ↓
前端保存 Token
  ↓
用户访问受保护接口
```

### 7.2 简历上传与解析流程

```text
用户登录系统
  ↓
进入简历管理页面
  ↓
上传 PDF / DOCX 简历
  ↓
后端校验文件类型和大小
  ↓
保存文件
  ↓
调用文件解析工具提取文本
  ↓
保存简历元数据和解析文本
  ↓
返回上传成功结果
  ↓
用户查看简历解析内容
```

### 7.3 岗位 JD 管理流程

```text
用户登录系统
  ↓
进入岗位管理页面
  ↓
填写公司名称、岗位名称、岗位描述、技能要求等内容
  ↓
系统保存岗位 JD
  ↓
用户查看岗位列表
  ↓
用户选择某个岗位进行 AI 分析
```

### 7.4 AI 匹配分析流程

```text
用户选择简历
  ↓
用户选择目标岗位 JD
  ↓
点击“开始分析”
  ↓
后端查询简历解析文本
  ↓
后端查询岗位 JD 内容
  ↓
检查 Redis 是否已有缓存结果
  ↓
如果有缓存，直接返回分析结果
  ↓
如果没有缓存，构造 Prompt
  ↓
调用大语言模型 API
  ↓
解析 AI 返回结果
  ↓
保存分析报告到数据库
  ↓
写入 Redis 缓存
  ↓
返回匹配分数、优势、短板、技能缺口和优化建议
```

### 7.5 投递记录管理流程

```text
用户查看 AI 分析报告
  ↓
决定是否投递该岗位
  ↓
创建投递记录
  ↓
设置初始状态：待投递 / 已投递
  ↓
后续根据实际进展修改状态
  ↓
记录笔试、面试、Offer 或被拒情况
  ↓
填写备注和面试复盘
  ↓
系统形成投递过程记录
```

## 8. 功能需求分析

### 8.1 用户认证模块

#### 8.1.1 注册功能

用户可以通过用户名、密码、邮箱注册账号。

功能要求：

1. 用户名不能为空；
2. 用户名不能重复；
3. 密码不能为空；
4. 密码需要加密存储；
5. 邮箱格式需要合法；
6. 注册成功后返回用户基础信息。

#### 8.1.2 登录功能

用户可以使用用户名和密码登录系统。

功能要求：

1. 校验用户名是否存在；
2. 校验密码是否正确；
3. 登录成功后生成 JWT；
4. 返回 Token 和用户基础信息；
5. 登录失败时返回明确错误信息。

#### 8.1.3 用户信息查询

用户登录后可以查看自己的个人信息。

功能要求：

1. 只能查看自己的信息；
2. 需要携带有效 Token；
3. 返回用户 ID、用户名、邮箱、学校、专业、年级等信息。

### 8.2 简历管理模块

#### 8.2.1 简历上传

用户可以上传自己的简历文件。

功能要求：

1. 支持 PDF 和 DOCX 文件；
2. 限制文件大小；
3. 校验文件类型；
4. 保存文件基础信息；
5. 自动解析简历文本；
6. 保存解析后的文本内容。

#### 8.2.2 简历列表查询

用户可以查看自己上传过的简历。

功能要求：

1. 只能查看自己的简历；
2. 按上传时间倒序排列；
3. 显示文件名、文件类型、上传时间、是否默认简历。

#### 8.2.3 简历详情查询

用户可以查看某份简历的详情。

功能要求：

1. 展示文件基础信息；
2. 展示解析后的文本内容；
3. 支持后续手动编辑解析文本。

#### 8.2.4 删除简历

用户可以删除不需要的简历。

功能要求：

1. 只能删除自己的简历；
2. 删除数据库记录；
3. 后续可扩展为同时删除文件存储中的文件。

#### 8.2.5 设置默认简历

用户可以将某份简历设置为默认简历。

功能要求：

1. 每个用户只能有一份默认简历；
2. 设置新的默认简历时，需要取消原默认简历；
3. 默认简历可用于快速 AI 分析。

### 8.3 岗位 JD 管理模块

#### 8.3.1 创建岗位

用户可以手动创建目标岗位 JD。

功能字段：

1. 公司名称；
2. 岗位名称；
3. 岗位类型；
4. 工作地点；
5. 来源平台；
6. 岗位链接；
7. 岗位描述；
8. 技能要求。

#### 8.3.2 岗位列表查询

用户可以查看自己保存的岗位。

功能要求：

1. 只能查看自己的岗位；
2. 支持按岗位类型筛选；
3. 支持按公司名称搜索；
4. 支持按创建时间排序。

#### 8.3.3 岗位详情查询

用户可以查看岗位详情。

功能要求：

1. 展示岗位基础信息；
2. 展示完整岗位 JD；
3. 展示是否已经进行过 AI 分析；
4. 展示是否已经创建投递记录。

#### 8.3.4 修改岗位

用户可以修改岗位信息。

功能要求：

1. 只能修改自己创建的岗位；
2. 修改岗位 JD 后，原有 AI 分析报告不自动覆盖；
3. 后续可以提示用户重新分析。

#### 8.3.5 删除岗位

用户可以删除岗位记录。

功能要求：

1. 只能删除自己的岗位；
2. 如果该岗位已关联投递记录，第一阶段可以逻辑删除；
3. 避免误删影响历史投递记录。

### 8.4 AI 分析模块

AI 分析模块是系统核心模块。

#### 8.4.1 简历与岗位匹配分析

用户选择一份简历和一个岗位 JD 后，系统生成 AI 分析报告。

输入：

1. resumeId；
2. jobId。

处理流程：

1. 查询简历解析文本；
2. 查询岗位 JD；
3. 构造 Prompt；
4. 调用 AI API；
5. 解析返回结果；
6. 保存分析报告；
7. 返回分析结果。

输出：

1. 匹配分数；
2. 匹配等级；
3. 简历优势；
4. 简历短板；
5. 技能缺口；
6. 简历优化建议；
7. 岗位准备建议。

#### 8.4.2 分析报告查询

用户可以查看历史 AI 分析报告。

功能要求：

1. 只能查看自己的分析报告；
2. 支持按岗位查询；
3. 支持按简历查询；
4. 支持按创建时间倒序排列。

#### 8.4.3 简历优化建议生成

系统可以根据简历内容和岗位 JD 生成优化建议。

建议内容包括：

1. 技能关键词补充建议；
2. 项目经历改写建议；
3. 简历表达优化建议；
4. 缺失经历补强建议；
5. 面试准备建议。

#### 8.4.4 AI 分析缓存

系统应对相同简历和相同岗位 JD 的分析结果进行缓存。

功能要求：

1. 使用 Redis 存储分析结果；
2. 缓存 Key 可由 userId、resumeId、jobId 组成；
3. 如果命中缓存，则直接返回；
4. 如果没有命中，则调用 AI API；
5. 缓存应设置过期时间。

### 8.5 投递记录模块

#### 8.5.1 创建投递记录

用户可以为某个岗位创建投递记录。

字段包括：

1. 岗位 ID；
2. 简历 ID；
3. 当前状态；
4. 投递日期；
5. 面试时间；
6. 备注。

#### 8.5.2 查询投递记录

用户可以查看自己的投递记录列表。

功能要求：

1. 只能查看自己的投递记录；
2. 支持按状态筛选；
3. 支持按公司名称搜索；
4. 支持按投递时间排序。

#### 8.5.3 修改投递状态

用户可以根据实际情况更新投递状态。

状态包括：

1. 待投递；
2. 已投递；
3. 笔试中；
4. 一面；
5. 二面；
6. HR 面；
7. Offer；
8. 被拒；
9. 放弃。

#### 8.5.4 投递备注与复盘

用户可以记录投递和面试过程中的备注。

内容包括：

1. 笔试内容；
2. 面试问题；
3. 面试表现；
4. 复盘总结；
5. 后续改进方向。

### 8.6 数据看板模块

数据看板属于第二阶段功能。

功能包括：

1. 投递总数；
2. 已投递数量；
3. 面试数量；
4. Offer 数量；
5. 被拒数量；
6. 平均匹配分数；
7. 常见技能短板；
8. 投递状态分布。

## 9. 非功能需求分析

### 9.1 安全性需求

系统涉及用户简历、个人信息和投递记录，需要保证基本安全性。

安全要求：

1. 用户密码必须加密存储；
2. 用户登录后通过 JWT 访问接口；
3. 用户只能访问自己的数据；
4. 文件上传需要校验类型和大小；
5. 防止未授权访问；
6. 防止越权查询；
7. AI API Key 不允许写死在代码中；
8. 敏感配置通过环境变量或配置文件管理。

### 9.2 性能需求

第一阶段系统访问量较小，但仍需考虑基本性能。

性能要求：

1. 普通查询接口响应时间应尽量小于 500ms；
2. AI 分析接口由于依赖外部模型，允许较长响应时间；
3. 相同简历和岗位组合应使用 Redis 缓存；
4. 文件解析过程应控制文件大小，避免占用过多资源；
5. 后续可将 AI 分析改为异步任务。

### 9.3 可维护性需求

项目需要保持清晰的代码结构，便于后续迭代。

可维护性要求：

1. 使用分层架构；
2. Controller 只负责接收请求和返回响应；
3. Service 负责业务逻辑；
4. Mapper 负责数据库访问；
5. DTO 用于接收请求；
6. VO 用于返回响应；
7. 使用统一响应结构；
8. 使用统一异常处理；
9. 使用清晰的包结构；
10. 编写必要注释和接口文档。

### 9.4 可扩展性需求

系统后续需要支持更多 AI 和求职相关功能。

可扩展方向：

1. 支持更多简历格式；
2. 支持简历在线编辑；
3. 支持多模型切换；

## 12. 核心业务实体关系

核心关系如下：

```text
User 1 —— N Resume
User 1 —— N JobDescription
User 1 —— N AnalysisReport
User 1 —— N ApplicationRecord
Resume 1 —— N AnalysisReport
JobDescription 1 —— N AnalysisReport
JobDescription 1 —— N ApplicationRecord
Resume 1 —— N ApplicationRecord
```

解释：

1. 一个用户可以上传多份简历；
2. 一个用户可以保存多个岗位 JD；
3. 一份简历可以和多个岗位进行匹配分析；
4. 一个岗位可以和多份简历进行匹配分析；
5. 一个岗位可以创建投递记录；
6. 投递记录可以关联某份简历。

## 14. 关键业务规则

1. 用户只能访问自己的简历、岗位、分析报告和投递记录；
2. 用户密码必须加密存储；
3. 简历文件必须限制类型和大小；
4. 每个用户可以上传多份简历；
5. 每个用户最多只能设置一份默认简历；
6. AI 分析必须基于用户自己的简历和岗位；
7. 相同简历和岗位组合可以复用缓存结果；
8. 投递记录状态必须在系统定义范围内；
9. 删除岗位时应考虑已有投递记录；
10. AI 原始返回结果应保存，便于后续排查。

## 15. 系统主要风险与应对方案

| 风险 | 说明 | 应对方案 |
|---|---|---|
| AI 返回格式不稳定 | 大模型可能不按预期返回 JSON | 固定 Prompt、格式校验、保存原始结果 |
| 文件解析失败 | PDF/DOCX 格式复杂 | 支持用户手动编辑解析文本 |
| API 调用成本高 | AI 接口可能收费 | 使用 Redis 缓存、限制调用次数 |
| 开发范围过大 | 功能过多导致延期 | 第一阶段只做 MVP |
| 用户数据越权 | 用户访问到他人数据 | 所有查询必须带 userId 条件 |
| 文件上传安全 | 恶意文件上传 | 限制文件类型、大小和存储路径 |
| 项目复杂度上升 | 后期功能变多难维护 | 保持分层架构和模块化设计 |

## 18. 系统分析结论

InternPilot 系统面向大学生实习求职中的真实痛点，结合简历解析、岗位 JD 管理、AI 匹配分析和投递记录管理，能够形成完整的业务闭环。

从技术角度看，系统采用 Spring Boot、Spring Security、JWT、MySQL、Redis 和大语言模型 API 等成熟技术，开发风险可控，适合个人独立完成。

从项目价值角度看，该系统既能解决实际问题，又能展示 Java 后端开发、权限认证、文件处理、缓存优化、AI 应用集成和工程化设计能力，适合作为软件工程学生寻找暑期实习时的主打项目。
